The problem with traditional Kafka consumers

Running Kafka consumers in production means dealing with a long tail of operational concerns that have nothing to do with your actual business logic.

PAIN POINT 01

Consumer group management

Rebalances, lag monitoring, offset commits — all require custom tooling and constant vigilance.

PAIN POINT 02

Scaling on spikes

Traffic spikes mean more consumers. More consumers mean coordination overhead and potential rebalance storms.

PAIN POINT 03

Infrastructure to maintain

EC2 instances, containers, health checks, deploy pipelines — just to run a poll() loop.

PAIN POINT 04

Retry logic

Dead letter queues, poison pill handling, backoff strategies — reimplemented for every service.

💡

The insight: Most Kafka consumers are stateless — they receive a batch, process it, and repeat. That's a perfect fit for Lambda's execution model. AWS ESM lets you replace the entire consumer infrastructure with a managed poller that invokes your function.

How Kafka Event Source Mapping works

ESM is an AWS-managed Kafka consumer that runs inside Lambda's infrastructure. You don't provision it, scale it, or operate it. You configure it — and it continuously polls your Kafka topic, batches messages, and invokes your Lambda function.

ESM maps one Lambda invocation per partition per batch. If your topic has 10 partitions and a batch is ready on 6 of them, Lambda fires 6 concurrent invocations. Parallelism is naturally bounded by your partition count — a clean, predictable scaling model.

What AWS manages for you

MANAGED

Consumer group lifecycle

ESM creates and manages a dedicated consumer group. Partition assignment and rebalancing are transparent.

MANAGED

Offset checkpointing

Offsets advance only after your function returns successfully. Errors cause a re-delivery of the same batch.

MANAGED

Retry with backoff

Failed batches are retried with exponential backoff. You configure max attempts and the on-failure destination.

MANAGED

Batching window

Configurable MaximumBatchingWindowInSeconds (0–300s) lets you trade latency for throughput.

Rule of thumb: Use on-demand for dev/staging and non-latency-sensitive workloads. Switch to provisioned for payment processing, fraud detection, or anywhere p99 latency matters. A 2–4 second cold start is invisible in a batch analytics pipeline but catastrophic in a real-time fraud check.

Anatomy of a Kafka ESM event

Lambda receives a single JSON payload per invocation, containing one or more topic-partition batches. Every message value is Base64-encoded — your first job is always to decode it.

{ "eventSource": "aws:kafka", "eventSourceArn": "arn:aws:kafka:us-east-1:123:cluster/prod/abc", "records": { // keyed by "topic-partition" "orders.created-0": [ { "topic": "orders.created", "partition": 0, "offset": 1042, "timestamp": 1705314720123, // Unix ms "key": "T1JELTIwMjQ=", // ← Base64 encoded "value": "eyJvcmRlcklkIjoi...", // ← Base64 encoded "headers": [ { "content-type": [97,112,112...] } // byte array ] } ] } }
🔑

Key + Value decoding: Buffer.from(msg.key, 'base64').toString('utf-8') — always do this before anything else. Never pass raw Base64 to your downstream logic.

📋

Headers decoding: Each header value is a byte array (number[]). Use Buffer.from(bytes).toString('utf-8'). Headers are great for routing metadata, trace IDs, and schema versions.

⏱️

Timestamp: This is the Kafka produce timestamp in Unix milliseconds. Wrap it in new Date(msg.timestamp).toISOString() for readable logs and audit trails.

Production handler walkthrough

The handler in the zip below follows a four-layer pattern: decode → enrich → route → observe. Here's the core loop with annotations:

src/handler.mjs Node.js ESM 
export const handler = async (event, context) => {
  // ① Iterate topic-partitions in the batch
  for (const [topicPartition, messages] of
       Object.entries(event.records)) {

    for (const message of messages) {
      // ② Decode Base64 key + value
      const key   = message.key
        ? Buffer.from(message.key, 'base64').toString('utf-8')
        : null;
      const value = message.value
        ? Buffer.from(message.value, 'base64').toString('utf-8')
        : null;

      // ③ Parse JSON — with a graceful fallback for raw strings
      let payload = null;
      try { payload = JSON.parse(value); }
      catch { payload = value; }

      // ④ Hand off to domain logic
      await processMessage({ ...message, key, payload });

      // ⑤ Emit CloudWatch EMF metric (no extra API call)
      metrics.increment('messages.processed', {
        topic: message.topic,
        partition: String(message.partition),
      });
    }
  }
};

Topic routing in processor.mjs

The processor uses a simple registry pattern. Adding support for a new topic is a two-line change — register a handler, write the function:

src/processor.mjs Node.js ESM 
// ① Register topic handlers here
const TOPIC_HANDLERS = {
  'orders.created':      handleOrderCreated,
  'payments.processed':  handlePaymentProcessed,
  'user.events':          handleUserEvent,
};

export async function processMessage(message) {
  const handler = TOPIC_HANDLERS[message.topic];
  if (!handler) {
    // Unknown topic → log and skip (don't throw)
    logger.warn('No handler for topic', { topic: message.topic });
    return;
  }
  await handler(message);
}

Configuring the event source mapping

ESM wires your MSK cluster to your Lambda function. Two ways: AWS CLI for one-off setup, Terraform for repeatable infrastructure.

AWS CLI

$ aws lambda create-event-source-mapping \ --function-name kafka-consumer-prod \ --event-source-arn arn:aws:kafka:us-east-1:123:cluster/prod/abc \ --topics orders.created,payments.processed \ --starting-position LATEST \ --batch-size 100 \ --maximum-batching-window-in-seconds 5 \ --destination-config '{"OnFailure":{"Destination":"arn:aws:sqs:...:kafka-dlq"}}'

Key parameters explained

PARAM

--batch-size

Max messages per invocation (1–10,000). Higher = more throughput. Lower = less latency per message.

PARAM

--maximum-batching-window

Seconds ESM waits to accumulate a full batch. Set to 0 for minimum latency. Set to 5–30 for higher throughput.

PARAM

--starting-position

LATEST = only new messages. TRIM_HORIZON = from the beginning of the retention window.

PARAM

--destination-config

Where to send failed batch metadata after all retries are exhausted. Always wire this to an SQS DLQ.

Production best practices

These aren't optional — each one has a failure mode behind it that has burned production systems.

01 · CRITICAL

Make handlers idempotent

ESM retries entire batches on failure. If your handler credits a payment twice, that's a production incident. Use a DynamoDB condition write or a Redis SET NX as a deduplication gate.

02 · CRITICAL

Always configure a DLQ

Without a destination config, exhausted retry batches are silently dropped. The DLQ captures the batch metadata (topic, partition, offsets) so you can replay or investigate.

03 · PERFORMANCE

Tune batch size to your p99

Start at 100. Load test with realistic payloads. If your timeout is 5 minutes and a 1,000-message batch takes 4.5 minutes, one slow message can cause the whole batch to time out.

04 · PERFORMANCE

Memory-size before concurrency

Lambda CPU is proportional to memory. Doubling memory from 512MB to 1024MB often halves duration — which halves cost at the same throughput. Profile before scaling concurrency.

05 · OBSERVABILITY

Use CloudWatch EMF for metrics

The metrics.mjs in the zip emits EMF-formatted JSON. CloudWatch parses it automatically — zero extra API calls, sub-second metric resolution, and no additional latency.

06 · SECURITY

Least-privilege IAM

The Lambda role needs only kafka-cluster:ReadData, DescribeTopic, DescribeGroup, and AlterGroup on the specific cluster ARN. The Terraform scopes this correctly.

Real-world use cases

ESM shines anywhere you need stateless, scalable, event-driven processing without the cost of always-on compute:

Real-time analytics pipelines

Aggregate clickstream events into time-windowed stats. Fan out to DynamoDB, S3, and Kinesis Data Firehose from a single Lambda.

Payment & fraud detection

Process payment events with sub-second latency. Apply ML model inference inline. Use Provisioned mode to eliminate cold-start variance.

Log ingestion & enrichment

Parse, enrich with geo/IP data, and route structured logs to OpenSearch or S3. Lambda scales linearly with log volume.

Event-driven microservices

Replace polling-based integrations. Each downstream service subscribes to relevant topics — fully decoupled, no shared databases.
